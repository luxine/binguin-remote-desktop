更新时间：2026-03-03

## 1. 目标与边界

本方案的目标是基于当前已完成品牌化改造的 `BinguinDesk`，搭建一套可长期在企业内部使用的远程桌面体系，满足以下要求：

- 使用企业自建的 `hbbs` / `hbbr` 服务器，不依赖 RustDesk 公共服务器。
- 使用企业自定义客户端，默认连接到企业自建服务器。
- 支持当前已经落地的客户端构建目标：`Windows x64` 与 `Ubuntu x64`。
- 以“稳定、可维护、便于升级”为第一优先级，不先做高风险的深度二开。
- 仅替换外显品牌，暂时不改内部服务名、协议名、进程名、配置目录名。

当前方案默认基于开源版 `rustdesk-server`（OSS）落地，不包含 `Pro` 的管理后台、地址簿管理、租户管理等高级能力。

## 3. 推荐的企业内部架构

建议第一阶段采用“单节点稳定版”架构，不要一开始就做多节点高可用。

### 3.1 一期推荐架构

- 一台 Linux 服务器同时运行：
  - `hbbs`：负责 ID 注册、打洞协调、信令
  - `hbbr`：负责中继转发
- 客户端为你当前仓库构建出的 `BinguinDesk`
- 被控端在 Ubuntu 上以 `systemd` 服务常驻运行
- 控制端可使用 Windows 或 Ubuntu 客户端

### 3.2 这样做的原因

- 单机部署变量最少，排障成本最低。
- `hbbs` 和 `hbbr` 同机时网络路径更简单。
- 先把“能稳定用”做成，再决定是否拆分服务、上多机或容灾。
- 你当前项目阶段更适合先完成“企业内部可用”，而不是一开始追求复杂架构。

### 3.3 二期可选扩展

在一期跑稳之后，再考虑：

- 主备服务器
- 跨地域节点
- 反向代理与证书统一管理
- 资产分组与发布系统
- 安全审计和连接日志归档

## 4. 自建服务器实施方案

本部分是当前最关键、但尚未完成的核心模块。

## 4.1 部署目标

部署一套企业自建的 `rustdesk-server`（开源版），包含：

- `hbbs`
- `hbbr`

并保证以下能力：

- 开机自启
- 异常退出自动恢复
- 密钥持久化
- 升级不丢配置
- 对外端口稳定可达

## 4.2 服务器准备条件

建议的最低配置：

| 项目 | 建议 |
| --- | --- |
| 系统 | Ubuntu 22.04 / 24.04 或 Debian 12 |
| CPU | 2 核起步，建议 4 核 |
| 内存 | 4 GB 起步，建议 8 GB |
| 磁盘 | 40 GB 起步，建议 80 GB+ |
| 网络 | 固定公网 IP 或可解析域名 |
| 时间同步 | 开启 `systemd-timesyncd` 或 `chrony` |
| Docker | 必装，建议使用 Docker Compose |

推荐准备：

- 一个稳定域名，例如 `desk.company.internal` 或 `desk.company.com`
- 服务器只做这一套远控服务，避免混跑过多无关业务
- 使用快照或云盘备份能力

## 4.3 网络与端口要求

RustDesk 当前默认端口常量在当前仓库中可见：

- `21116`：Rendezvous
- `21117`：Relay
- `21118`：WebSocket Rendezvous（可选）
- `21119`：WebSocket Relay（可选）

建议开放策略：

| 端口 | 协议 | 是否必须 | 用途 |
| --- | --- | --- | --- |
| 21115 | TCP | 建议 | 某些文档和部署场景会一起开放，保留兼容性 |
| 21116 | TCP | 必须 | 客户端与 `hbbs` 主通信 |
| 21116 | UDP | 必须 | NAT 测试 / 打洞相关 |
| 21117 | TCP | 必须 | `hbbr` 中继转发 |
| 21118 | TCP | 可选 | WebSocket 模式 |
| 21119 | TCP | 可选 | WebSocket Relay |
| 21114 | TCP | 可选 | 仅当你后续需要 API / Pro / 管理后台时再考虑 |

如果服务器在云上，需要同时检查：

- 云平台安全组
- 系统级防火墙（`ufw` / `firewalld`）
- 上游网络设备（如有）

## 4.4 官方服务端与当前客户端的关系

你当前这个仓库是客户端，不是服务端。  
真正要部署的服务端是独立项目 `rustdesk-server`。

官方参考：

- https://rustdesk.com/docs/en/self-host/
- https://rustdesk.com/docs/en/self-host/rustdesk-server-oss/docker/
- https://github.com/rustdesk/rustdesk-server

当前客户端代码里也明确说明它是通过 `rendezvous_mediator` 与 `rustdesk-server` 通信。

## 4.5 推荐部署方式：Docker Compose

第一阶段最稳的方式是 Docker Compose，使用宿主机网络。

在服务器上执行：

```bash
sudo apt update
sudo apt install -y ca-certificates curl gnupg
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo $VERSION_CODENAME) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
sudo systemctl enable --now docker
```

创建部署目录：

```bash
sudo mkdir -p /opt/rustdesk-server
sudo chown $USER:$USER /opt/rustdesk-server
cd /opt/rustdesk-server
mkdir -p data
```

创建 `compose.yml`：

```yaml
services:
  hbbr:
    container_name: hbbr
    image: rustdesk/rustdesk-server:latest
    command: hbbr
    network_mode: "host"
    volumes:
      - ./data:/root
    restart: unless-stopped

  hbbs:
    container_name: hbbs
    image: rustdesk/rustdesk-server:latest
    command: hbbs -r desk.company.com:21117
    network_mode: "host"
    volumes:
      - ./data:/root
    depends_on:
      - hbbr
    restart: unless-stopped
```

其中：

- `desk.company.com` 替换为你的真实域名或公网 IP
- `./data:/root` 用来持久化密钥和服务数据
- `restart: unless-stopped` 保证异常退出后自动恢复
- `network_mode: "host"` 在 Linux 上更直接，网络排障最简单

启动服务：

```bash
cd /opt/rustdesk-server
docker compose up -d
docker ps
docker logs hbbs --tail 100
docker logs hbbr --tail 100
```

## 4.6 服务器部署完成后必须做的检查

服务拉起后，必须逐项确认：

1. 容器状态正常

```bash
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
```

2. 端口监听正常

```bash
sudo ss -tulpen | grep -E '21115|21116|21117|21118|21119'
```

3. 日志无持续报错

```bash
docker logs hbbs --tail 200
docker logs hbbr --tail 200
```

4. 外网能访问端口

可在外部机器验证：

```bash
nc -vz desk.company.com 21116
nc -vz desk.company.com 21117
```

5. 至少两台测试客户端可以成功互联

- 同一内网环境测试
- 不同网络环境测试
- 强制 relay 场景测试

## 4.7 公钥与密钥管理

这是最重要的资产之一。

首次启动后，`/opt/rustdesk-server/data` 下通常会生成：

- `id_ed25519`
- `id_ed25519.pub`

其中：

- `id_ed25519`：私钥，严禁泄露
- `id_ed25519.pub`：公钥，需要发给客户端

必须执行的动作：

- 立即备份整个 `data/` 目录
- 至少做一份离线备份
- 记录该公钥对应的是哪一套生产环境
- 不要轻易删除或重建 `data/` 目录

如果密钥丢失并重新生成：

- 服务器身份会变化
- 已发出去的客户端若绑定了旧公钥，需要重新配置或重新发包

## 4.8 服务器升级流程

建议固定升级流程，不要直接在线随意改。

标准升级步骤：

```bash
cd /opt/rustdesk-server
docker compose pull
docker compose up -d
docker ps
docker logs hbbs --tail 100
docker logs hbbr --tail 100
```

升级前必须做：

- 备份 `data/`
- 记录当前镜像版本
- 保留最近一个可回退的版本标签

回滚方式：

- 改回上一版镜像 tag
- `docker compose up -d`

## 4.9 服务器安全与稳定建议

建议立即落实以下规则：

- 仅开放必要端口，不暴露无关服务
- 限制 SSH 来源 IP
- 为服务器启用自动安全更新
- 系统开启时间同步
- 定期查看 Docker 日志
- 为宿主机设置基础监控：CPU、内存、磁盘、网络
- 使用独立域名，避免未来迁移时改大量客户端配置

## 5. 自定义客户端实施方案

本部分决定“用户安装后是否能直接用企业服务器”，是第二个核心模块。

## 5.1 当前已完成的客户端工作

你目前已经完成：

- 品牌外显替换为 `BinguinDesk`
- Windows / Ubuntu 的最小化构建链
- 安装包对外名称调整
- Ubuntu 被控端部署文档整理

这说明“企业品牌客户端外壳”已经具备基础交付能力。

## 5.2 当前还未完成的客户端关键项

当前仍未落地的关键点：

- 默认服务器仍未改到你的企业服务器
- 默认公钥仍未改到你的企业公钥
- 默认 API 行为仍保留上游逻辑
- 尚未形成“安装即连内网服务器”的最终安装包

## 5.3 当前客户端默认连接逻辑

当前代码里仍然保留了 RustDesk 公共服务器与公共公钥默认值：

- `RENDEZVOUS_SERVERS = ["rs-ny.rustdesk.com"]`
- `RS_PUB_KEY = ...`

这意味着如果你不做进一步处理，客户端仍有可能落回公共默认路径，而不是完全绑定企业环境。

## 5.4 客户端需要修改的配置项

企业内使用时，最关键的客户端配置项是：

| 配置项 | 作用 | 是否建议配置 |
| --- | --- | --- |
| `custom-rendezvous-server` | 企业自建 `hbbs` 地址 | 必须 |
| `key` | 企业服务器公钥 | 必须 |
| `relay-server` | 企业 `hbbr` 地址 | 建议 |
| `api-server` | 企业 API 地址 | 可选，OSS 可先留空 |

这些配置项在当前代码里都是已支持的。

## 5.5 推荐做法：先做“企业默认配置”，不要先改底层标识

为了降低风险，建议分两层推进：

### 层级 A：必须做

让安装后的客户端默认连接企业服务器：

- 预置 `custom-rendezvous-server`
- 预置 `key`
- 视情况预置 `relay-server`

这样可以先达成“安装即用”。

### 层级 B：暂不做

暂时不要改这些内部标识：

- 进程名
- 服务名
- 配置目录名
- 协议名
- 包内内部路径

原因：

- 这会影响升级兼容性
- 会扩大排障范围
- 会增加 Windows / Linux 双平台维护成本

## 5.6 客户端有两种实现方式

### 方式一：修改默认常量（简单直接）

直接把默认服务器与默认公钥改成企业值。

需要调整的核心位置：

- `libs/hbb_common/src/config.rs`

需要替换的内容：

- `RENDEZVOUS_SERVERS`
- `RS_PUB_KEY`

优点：

- 最直接
- 产物安装后天然指向企业环境

缺点：

- 后续换域名或换密钥时，需要重新发版

### 方式二：使用预置配置 / 自定义客户端配置（更可维护）

当前代码支持读取自定义客户端配置。  
推荐用它来写入：

- `custom-rendezvous-server`
- `relay-server`
- `key`
- `api-server`
- 其他企业默认策略

优点：

- 更灵活
- 不必频繁改底层常量
- 更适合后续扩展企业策略

缺点：

- 需要把预置配置注入到构建或安装流程中

## 5.7 API 逻辑的一个注意点

当前代码逻辑中，如果设置了自定义服务器，客户端会尝试推导一个默认 API 地址，通常会落到：

- `http://<你的服务器>:21114`

这对开源版场景不一定是问题，但需要明确：

- 如果你暂时不用 Pro/API，可先不依赖它
- 如果你想彻底避免任何 21114 访问，需要后续单独收敛这段逻辑
- 如果你将来要接企业 API，这段逻辑可以继续利用

## 5.8 建议的客户端改造顺序

建议按下面的顺序做：

1. 先把企业服务器、公钥写入默认客户端配置
2. 构建出新的 Windows / Ubuntu 安装包
3. 用 2 到 5 台测试机验证连通性
4. 再决定是否需要进一步隐藏不必要的设置项
5. 最后才考虑深度改名或内部逻辑裁剪

## 6. 被控端部署与常驻方案

这部分你已经有单独文档，但这里整理为整体方案中的标准要求。

根目录已有文档：

- `Linux被控端部署与升级手册.md`

在企业内实际执行时，建议统一采用以下标准：

## 6.1 部署方式

- Ubuntu 使用 `.deb` 安装
- 以 `root` 权限运行服务
- 使用 `systemd` 常驻
- 开机自启

## 6.2 可用性要求

必须保证：

- 进程异常退出可自动拉起
- 主机重启后自动恢复服务
- 网络恢复后能够继续被连接
- 禁止休眠、自动挂起、自动锁屏影响无人值守

## 6.3 图形环境建议

Linux 被控端如果需要长期稳定的桌面控制，建议：

- 优先使用 `Xorg`
- 配置自动登录
- 关闭自动锁屏
- 减少 Wayland 兼容性变量

## 6.4 运维要求

建议把这部分纳入标准运维清单：

- 服务状态巡检
- 日志保留
- 定期升级
- 回滚预案
- 统一配置备份
